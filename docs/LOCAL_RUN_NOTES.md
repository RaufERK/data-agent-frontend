# Локальный запуск проекта

## Краткий вывод

Проект можно запускать локально, но frontend сам по себе не раскрывает основную функциональность. Почти все важные сценарии завязаны на backend через `/api`.

Основные данные проект не берёт из внешнего облака автоматически. Базовый сценарий рассчитан на то, что пользователь сам загружает Excel/CSV-файлы, backend сохраняет их в локальную сессию/DuckDB, а frontend отображает результат через API.

## Что запустится локально

- Frontend можно запустить отдельно через Vite.
- Backend можно поднять локально вместе с PostgreSQL.
- Полный локальный запуск предусмотрен через Docker Compose из backend-репозитория.
- Базовый сценарий с загрузкой своих файлов должен быть достижим локально.

## Что не будет работать или может работать ограниченно

- Frontend без backend почти сразу упрётся в `/api`: авторизация, загрузка файлов, качество данных, чат, дашборды и квоты требуют backend.
- Demo-сценарий ожидает Excel-файлы в `public/datasets`, но сейчас в frontend-репозитории `public` пустой. Без этих файлов встроенные demo-данные могут не загрузиться.
- LLM/chat/dashboard generation/vision могут не работать полноценно без внешних настроек и ключей: `LITEPROXY`, `GPT2GIGA`, GigaChat и похожих сервисов.
- BI exports в DataLens, Visiology и Foresight локально, скорее всего, не заработают без credentials и доступов к внешним системам.

## Рекомендуемый локальный запуск

Запускать полный проект лучше из backend-репозитория:

```bash
cd /Users/rauferk/Documents/WEB_PROJ/tot/data-agent-backend
cp .env.example .env
docker compose up --build
```

После запуска frontend должен быть доступен на:

```text
http://localhost:5000
```

Проверка API:

```bash
curl http://localhost:5000/api/health
```

## Итог для дальнейшей работы

Локально проект поднять можно. Для разработки UI и базового сценария загрузки файлов этого должно быть достаточно. Но для полноценной демонстрации всех возможностей нужны дополнительные вещи:

1. Backend + PostgreSQL.
2. Demo Excel-файлы в `public/datasets`, если нужен встроенный demo-flow.
3. Внешние LLM/vision credentials, если нужен чат, генерация дашбордов и анализ изображений.
4. Credentials для BI-систем, если нужно проверять export-интеграции.

Приоритет для локальной проверки: сначала поднять Docker Compose, проверить `/api/health`, затем попробовать авторизацию и загрузку собственного Excel/CSV-файла.
